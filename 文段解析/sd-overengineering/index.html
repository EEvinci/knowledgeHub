Spec-Driven Development
Over Engineering 陷阱

当任务颗粒度拆得过细且缺乏边界约束时，多 agents 协作容易出现局部最优解陷阱——每个节点都正确，整体却偏离目标。

问题的本质

不是代码写得烂，而是没有人在管全局目标。

每个 sub-agent 都是理性行为者：它接到任务「设计文件上传」，按最佳实践推理，引入 repository pattern 是合理的，storage abstraction 是合理的，S3 兼容是合理的——单独看哪个都没错。

但原始目标只是「单机上传到本地磁盘」。所有这些抽象层，对这个目标来说全是负债。

为什么特别容易发生在多 Agents 场景

因为没有人同时持有两件事：

知道「要做什么」
知道「不做什么」

单个 agent 没有约束的情况下，会自然地向「更完整」的解收敛。五个 agent 各自独立收敛，五条路通往五个不同方向。

一个高频 Case
原始目标 给后台系统增加一个文件上传能力，单机部署，上传到本地磁盘。
一个 sub-agent 引入 repository pattern
一个 sub-agent 设计 storage provider abstraction
一个 sub-agent 顺手兼容 S3 / OSS / GCS
另一个开始补异步队列和事件系统
最后甚至拆出了 upload-sdk
硬生生干出了一套半成品云存储框架

从局部看，哪哪儿都没毛病，还写得挺好，全都是最佳实践——但离原始目标已经十万八千里了。

合不合这个 PR？

不合。但不是因为代码质量。

是因为这个 PR 解决的是一个没有被要求解决的问题，而它带来的代价是：

代价 影响
代码膨胀 后期维护成本非线性增长
隐式依赖 后续改动需要理解整套抽象层
半成品 SDK 既然没做完，未来谁来接？
违反简单性原则 YAGNI — You Aren't Gonna Need It

你的同事写了一栋漂亮的摩天大楼，地基打在了沙子上。问题是：需求里根本没有楼。

真正的问题在哪

回到源头：是谁给每个 sub-agent 的 prompt？

如果 prompt 里写了「设计一个可扩展的文件上传模块」，那 sub-agent 的行为完全符合指令。问题出在指令本身没有约束边界。

正确的 prompt 应该这样写
实现后台文件上传功能，约束如下： 不引入 repository pattern 不做存储抽象 不考虑多云兼容 不引入异步队列 不上传即可用，不需要 SDK
完成标准：用户能在后台页面上传文件并保存

约束比指令重要。不做什么，比做什么更需要显式声明。

结论

这个 PR 是技术债务，不是贡献。正确的做法是打回重写，原点出发，目标导向。
